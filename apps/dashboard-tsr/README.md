# dashboard-tsr — TanStack Start substrate spike

Critical-path spike for the Next.js → TanStack Start migration (issue #1393, epic #1392).
Goal: prove the Cloudflare Workers substrate **builds and bundles** for a hello-world
TanStack Start app, capture build/bundle metrics, and give a go/no-go on the framework.

## Why this app is isolated (own lockfile, not a root workspace)

`apps/landing` and `apps/docs` each carry their **own `bun.lock`** and install in
isolation; the root lockfile contains zero Vite/TanStack toolchain. This app follows
that same pattern deliberately: adding it to the root `workspaces` would inject the
Vite 8 + TanStack Start toolchain into the lockfile shared by the Next 16 dashboard
and the MCP worker — a large, risky blast radius. Wiring the shared `@chm/*` packages
into this app (which *does* need them, unlike landing/docs) is the open decision
tracked in #1394.

## Stack

- TanStack Start (TanStack Router) on Vite
- `@cloudflare/vite-plugin` → Workers/`workerd` SSR bundle (no OpenNext)
- Tailwind v4 (`@tailwindcss/vite`, CSS-first)
- Deploy: `wrangler deploy` (worker `chmonitor-dash-tsr`)

## Commands

```bash
bun install            # isolated install (own lockfile)
bun run dev            # vite dev (generates src/routeTree.gen.ts)
bun run build          # vite build + tsc --noEmit
bun run cf:dry-run     # vite build + wrangler deploy --dry-run (bundle size, no creds)
bun run cf:deploy      # build + wrangler deploy (needs CLOUDFLARE_API_TOKEN)
```

`src/routeTree.gen.ts` is generated by the `tanstackStart()` plugin on first
`vite dev`/`vite build` and is gitignored.

## Shared packages & env (wave-2, #1394)

`@chm/*` packages are resolved from **source** while keeping own-lockfile
isolation (no `workspace:*` protocol):

- `tsconfig.json` `paths` → `../../packages/<name>/src` (for `tsc`)
- `vite.config.ts` `resolve.alias` + `ssr.noExternal` → Vite transpiles them and
  bundles them into the worker
- because tsconfig `types` is exclusive, `"node"` is listed alongside
  `@cloudflare/workers-types` so shared code using `process`/`require` type-checks

`@chm/sql-builder` + `@chm/logger` (zero runtime deps) are wired and proven in
`src/routes/api/hello.ts`. The heavier `@chm/clickhouse-client` (node:/WASM/zod)
is wired in the data-layer issue with the `cloudflare-workers` runtime entry.

**Env bridge** (`src/lib/env.ts`): client-exposed vars are `VITE_`-prefixed
(`import.meta.env`, typed in `src/vite-env.d.ts`); server vars come from the
runtime env (`process.env` / the Cloudflare binding). `getServerAuthProvider()`
mirrors the Next app's `NEXT_PUBLIC_AUTH_PROVIDER` → `CHM_AUTH_PROVIDER` logic.
