# Contributing to clickhouse-monitoring

Thanks for helping improve this project. This guide covers the essentials for
getting a working development environment, running checks, and sending a pull
request.

---

## Prerequisites

- [Bun](https://bun.sh) — the only package manager used in this repo. The
  `preinstall` hook blocks `npm` and `pnpm`.
- Node.js 20+ (for scripts that don't run under Bun directly).
- A ClickHouse instance for integration tests (optional for most work).

---

## Setup

```bash
git clone https://github.com/chmonitor/chmonitor
cd clickhouse-monitoring
bun install
```

`bun install` installs root-workspace packages and runs Husky hooks (`prepare`).
The dashboard (`apps/dashboard`) has its own lockfile and must be installed
separately when working on it:

```bash
cd apps/dashboard
bun install --frozen-lockfile
```

## Monorepo layout

```
apps/
  dashboard/   TanStack Start app → dash.chmonitor.dev (primary app)
  docs/        Astro Starlight docs site → docs.chmonitor.dev
  landing/     Astro marketing site → chmonitor.dev
  mcp/         Cloudflare Worker MCP server
packages/
  clickhouse-client/  ClickHouse HTTP client (shared)
  logger/             Structured logger (shared)
  mcp-server/         MCP server core (shared)
  platform/           Platform utilities (shared)
  sql-builder/        SQL builder (shared)
  types/              Shared TypeScript types
```

All apps use `bun` as the package manager. `apps/docs`, `apps/landing`, and
`apps/mcp` have their own `bun.lock` and are built independently.

## Development commands

Run from the **repo root** unless noted.

| Command | What it does |
|---------|-------------|
| `bun run dev` | Start all apps in dev mode (turbo) |
| `bun run build` | Build everything (turbo) |
| `bun run lint` | Biome lint |
| `bun run fmt` | Biome format |
| `bun run check` | Biome full check (lint + format) |
| `bun run type-check` | TypeScript type check (turbo) |
| `bun run test` | Run all tests (turbo) |
| `bun run test:unit` | Dashboard unit tests only |
| `bun run test:coverage` | Unit tests + coverage |
| `bun run test:e2e:headless` | Cypress e2e tests (headless) |
| `bun run depcruise` | Validate dependency boundaries |

For dashboard-specific commands:

```bash
cd apps/dashboard
bun run build         # Vite build + tsc --noEmit
bun run dev           # Vite dev server
bun run type-check    # tsc --noEmit (src only)
bun run type-check:test  # tsc -p tsconfig.test.json (see note below)
bun run test          # Bun test suite
bun run cf:dry-run    # Build + wrangler dry-run (bundle size check)
```

## Important CI notes

### Type-check gap (tsconfig.test.json)

The root `bun run build` and `bun run type-check` check `src/` only via
`tsconfig.json`. CI runs a **separate** `bun run type-check:test` step (using
`tsconfig.test.json`) to catch type errors in `*.test.ts` files. These errors
pass the normal build but fail CI.

**Always run `bun run type-check:test` before opening a PR** if you touched
test files.

### Generated route tree

`apps/dashboard/src/routeTree.gen.ts` is gitignored and generated by Vite on
`bun run build`. TypeScript cannot resolve TanStack route types without it.
Type-checking test files in CI therefore runs **after** the Vite build (see
`cloudflare.yml`). If you see `Cannot find module '~react-router/...'` locally,
run `bun run build` first.

### Bundle size budget (CI: `bundle-size` workflow)

The dashboard deploys to Cloudflare Workers. The free-plan gzip limit is 3 MiB.
Baseline is ~1.82 MiB. CI measures the bundle on every push/PR via
`wrangler deploy --minify --dry-run` and warns above 2.5 MiB. See
`docs/knowledge/worker-bundle-size.md` for details and the budget constants in
`.github/workflows/bundle-size.yml`.

## Commit convention

This repo uses [Conventional Commits](https://www.conventionalcommits.org/).

```
<type>(<scope>): <short summary>

[optional body]

Co-Authored-By: duyetbot <bot@duyet.net>
```

**All commits must include the `Co-Authored-By` trailer above** (copy/paste it
exactly). PRs are squash-merged, so the trailer only needs to appear in the
final commit or PR description.

Common types: `feat`, `fix`, `chore`, `refactor`, `docs`, `perf`, `test`, `ci`.

Scope examples: `dashboard`, `mcp`, `landing`, `docs`, `deps`, `ci`.

Keep the summary line under 72 characters. No period at the end.

---

## Pull requests

- Open PRs against `main`.
- PR titles follow the same Conventional Commits format.
- CI must pass: `build`, `lint`, `dashboard` (Cloudflare deploy), and
  `unit-tests`. The `e2e-test`, `e2e-test-tsr`, `component-test`, and
  `unit-tests` (non-required) jobs are informational and do not block merge.
- The `bundle-size` and `axe-core` (accessibility) checks are non-required:
  they annotate the PR but do not block merge.
- Auto-merge is enabled for most PRs — check CI and fix failures before
  pushing follow-up commits (pushing after arming `--auto` strands
  mid-flight commits; see memory note `feedback-automerge-followup-race.md`).

## Releases

Releases are managed by [release-please](https://github.com/googleapis/release-please).
Do **not** manually bump `package.json` versions or edit `CHANGELOG.md`. A
`chore(main): release X.Y.Z` PR is opened automatically; leave it for the
maintainer to merge.

---

## Accessibility

The `a11y` CI workflow runs `@axe-core/cli` against WCAG 2.0 A/AA rules on a
set of key dashboard routes. It is currently non-required (informational only).
When adding new UI, check for missing `aria-label`, landmark roles, and
sufficient color contrast.

**i18n stance**: The dashboard is English-only for now. No i18n framework is
planned. Do not add translation infrastructure (`i18next`, `react-intl`, etc.)
without first discussing it in an issue.

---

## Docs

User-facing docs live in `docs/content/` and are published via `apps/docs`
(Astro Starlight). Edit only `docs/content/**` — `apps/docs/src/content/docs/`
is gitignored and regenerated at build time.

If you add, rename, or remove an agent tool, a skill, or an agent env var,
update `docs/content/ai-agent.mdx` in the same PR.
